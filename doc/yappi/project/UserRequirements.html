User Requirements

To find out the goals of this project we define User Requirements. These are small, separated tasks which can be implemented step by step. Every user requirement is one feature of the upcoming software and implementation. It is very important, that it is possible to check afterwards, if a particular user requirement was accomplished or not. If not, it is easier to define the future work, which has to be done.

We begin with the most general user requirement: A database has to be built up.

1)      Define a data model fitting to all data requirements.

See the data model section.

2)      Choose a database system which is accessible in many ways (also over the web) and run able on many platforms

Fitting to the very wide usage of the database we took XML as format. We have the following reasons: A standard database system like Oracle or MySQL is system dependent and needs a high level administration team. The advantage of such a system would be very fast access and a high security level. Both we don’t need really, because the data is most times just read only. The data changes not very often and could be just copied as a backup to a tape. There is no need of a high-tech database system.

A XML file can be published easy on the web without a special protocol or server. On the other hand a XML file can be also stored on a harddisk. You need no software for a XML file, but your XML interface.

XML is accessible in many standard database systems (for example Oracle) and in nearly every programming language (C++, Java, …).

3)      Create a human and computer readable database definition file.

As database definition standard we use the “XML Schema” definition. Unfortunately until now it is only a working draft and not a recommendation. 

August 16, 2000: Actual version of XML-Schema: 7 April 2000

4)      Create a test data file by hand

5)      Additional data must be easily addable

If a user wants to add his own data he can add a XML file with his own data. The data inside will be added.

6)      Data should be easily modifiable / overwrite able

If a user doesn’t want to use specific data or overwrite special values he can use a second XML file, where the specific data is provided.

7)      Data should be imported from other databases.

This database is not intended to be used as first order database. The intention is to provide the particle properties to a wide range of users, but the data itself should be taken out of a other database (Particle Data Group (PDG)). For this reason interfaces and import utilities must be written.

August 16, 2000: We have an import utility for the “.mc” file from the Particle Data Group Web Site (http://pdg.lbl.gov). Problem here is the contents: This is only very few of data: ASCII-Name, PDG-ID-Number, Mass, Width and Charge.

August 16, 2000: We have a slight approach to get the whole PDG-database. If this happens we have to look, what actually is in the database and how we can get the useful information out of it. If we get no access to this database another idea would be to get the information out of the web-published Postscript files (*.PS) on the PDG web site. I have looked a bit on it and it seems to be possible to get it out.

August 24, 2000: We have a copy of the PDG database from 1998, but the computed results are missing (table RESULTS_SUMMARY). I (Patrick Hellwig) have requested the FORTRAN programs to compute these values from the PDG. I hope they will arrive soon.

8)      A data administration program should be written.

The following idea: Build up an overall software: Add/Modify/Delete Particles, Add/Modify/Delete FamilyStructures, Add/Modify/Delete Sources, Add/Modify/Delete Comments, create new XML files (import or type in data), export data (create hardcode), search and browse data, display treelike structure, etc.

August 24, 2000: Until now it seems to me, that I will not have the time to do it until my departure.

9)      A Java Servlet should be written to browse and search through the database

August 16, 2000: The servlet can actually search for particle name and PDGID. But it can’t display the decay modes until now.

August 24, 2000: The servlet is now able to display also the decay modes either in HTML-format or LaTeX format

10)  The interface must be capable to read multiple data files

August 16, 2000: This must be also implemented.

August 24, 2000: Implementation has been done.

11)  Provide an web site with all the information

You are currently looking on it.

August 24, 2000: I have no specific server until now.

12)  The database must provide the same information like all the other old software databases

13)  It must be possible to implement new Particle Properties without recompiling code or changing the database structure

This is done by the data structure. You have an entry like: <data name=”Mass” value=”1.233”/> to store the data. So if additional data will come up it is easy to add another <data …/> element.

14)  For each data entry there should be the possibility for a source entry

August 16, 2000: I have thought about this, but it is not implemented yet.

August 24, 2000: The XML Schema file (XSD) is prepared for this, but the Parser doesn’t care about it. Until now I haven’t implemented the Interface for “source” entries.

15)  For each particle the following data must be stored:

  • Family (Bosons, Mesons, etc.)
  • Name in ASCII encoding
  • Name in typsetting format (LaTeX)
  • Standard Particle ID
  • Charge
  • Width (+Error, -Error)
  • Mass (+Error, -Error)
  • has antiparticle (yes / no)
  • 2J (Quantum Information)
  • <add more general data>
